# ADR 0013 — Dev inspector is `__DEV__`-only, not runtime-gated

**Status:** Accepted

A new in-page Daemon dev inspector (per-Daemon footer under each `.ai-panel` showing in-flight pip + last-round tool calls + token/cost line + active complications, a god-view 5×5 map between `#panels` and `#composer` with click-to-focus cone tinting, and a game-global strip above the map) is being added to surface engine-side and LLM-side per-Daemon state that's currently only visible via `console.log` (`src/spa/game/browser-llm-provider.ts:78`, `:90`) and the hidden `#action-log` aside (`src/spa/index.html:187`). The question this ADR records is *how* the inspector is gated: the existing dev surfaces in the codebase split between build-time (`__DEV__` from `src/spa/env.d.ts:11`, set by esbuild `define`, tree-shaken from prod) and runtime (`?debug=1` URL param at `src/spa/routes/game.ts:1137`, ships in every build, anyone can flip it on). Both patterns are already established; the inspector picks one.

## Decision

The inspector is gated by `__DEV__` only. There is no runtime flag (no `?debug=1`, no `?dev=1`, no localStorage key). In production builds the inspector markup, render code, and event-loop wiring are tree-shaken out entirely — the bundle is unchanged in size and behaviour from a non-inspector build.

This is a tuning surface, not a live-ops surface. It exists so the developer can iterate on prompts, personality synthesis, complication pacing, and content-pack tuning by watching three Daemons in parallel; it is not a tool for diagnosing a live player's report from production. Diagnosing live production is a separate problem with a separate (currently unsolved) surface; deliberately conflating the two would have meant shipping the inspector into prod and accepting both the bundle cost and the "anyone with the URL knows the flag" exposure for no tuning benefit.

As part of this work, the existing `?debug=1` `#action-log` aside (`src/spa/index.html:187`, toggle at `src/spa/routes/game.ts:1137-1144`, event handler at `:1461-1468`, styles at `src/spa/styles.css:713`) is deleted — the inspector supersedes it. The two `__DEV__`-gated `console.log` lines in `browser-llm-provider.ts` (`[cache]` cache-hit %, `[tools]` per-turn tool-name array) are also deleted; both signals now have a UI home inside the inspector footer.

## Considered Options

**`__DEV__` only, no runtime flag (chosen).** The inspector ships only in dev builds. Pure build-time gating — esbuild `define` makes `__DEV__` a literal `false` in prod, the dead branches drop out, and the prod bundle never carries inspector markup or render code. Matches the framing of the surface as a tuning loop, not a live-ops loop. Closes the door on "let me flip it on for one player" requests by making them impossible, which is the right answer when live-ops diagnostics deserves a purpose-built surface rather than a re-purposed tuning panel.

**`?debug=1` only, runtime gating in every build (rejected).** The inspector would ship in prod, hidden behind a URL flag. Rejected because the bundle cost is non-trivial (footer markup + map renderer + the wiring for in-flight pip edges + tooltip code), because the diagnostic value in prod is low (a live player can't usefully run `?debug=1` against their own session, and the dev can't reach into the player's browser to read it), and because once it ships in prod the temptation to make it a partial-live-ops tool grows — at which point its design centre-of-gravity drifts away from "iterate on prompts" toward "explain a bug report," and neither role gets served well. The right shape for a live-ops tool (auth-gated, audit-logged, read-only view of a player's session via shared link) is different enough from a tuning panel that they shouldn't share an implementation.

**Both, AND'd: `__DEV__` build *and* `?debug=1` flag (rejected).** Strictest possible — must be a dev build *and* have the flag. Rejected because the second gate adds nothing on top of the first: a dev build already implies the developer is the one looking, and requiring a URL flag inside a dev build is friction without payoff. The inspector should be visible by default in dev so we don't have to remember to enable it.

**Both, OR'd: `__DEV__` auto-on, `?debug=1` opt-in in prod (rejected).** The most-permissive option — visible by default in dev, available behind a flag in prod. Same rejection reasoning as the `?debug=1`-only option: it forces the inspector to be designed to work in both worlds, which compromises both. Either it's the tuning panel (and the prod opt-in path is unused) or it's the live-ops panel (and the dev-default path is over-equipped for tuning).

**New mechanism: separate localStorage `hi-blue:dev` key or `?dev=1` URL param (rejected).** Would create a third dev-gating mechanism alongside the existing `__DEV__` and `?debug=1`. Rejected on cohesion grounds — three ways to mean "show me dev stuff" is worse than two ways, and the existing two are already being collapsed (the `?debug=1` aside is being removed by this work).

**Repurpose `?debug=1` to mean "show inspector" rather than deleting it (rejected).** Would have preserved the URL surface for muscle memory. Rejected because the URL would then carry the same in-prod exposure as the standalone `?debug=1` option, defeating the point of the build-time gate. The URL-param channel and the build-time channel are not equivalent and shouldn't be made to alias.

## Consequences

- The inspector is *unreachable* in production builds. A player report that "this Daemon is doing something weird" cannot be answered by asking the player to flip a flag and screenshot the inspector. The answer to those reports continues to be: reproduce locally with the same seed / API key / save, then use the inspector. The seed-and-affordance test path (`?seed=…`, `?winImmediately=1`, etc., at `src/spa/routes/game.ts`) remains the production-side reproduction lever and is unaffected.
- If a live-ops diagnostic surface is ever needed (the open problem this ADR explicitly defers), it is a new design rather than a refit of the inspector. The architectural shape is different — live-ops needs auth, needs to be read-only against someone else's session, needs audit logging, needs deliberate redaction of any prompt content that could leak between sessions — and trying to evolve the tuning inspector into that role would force every property onto a surface that didn't need them.
- The `#action-log` aside and the two `console.log` lines in `browser-llm-provider.ts` are deleted in the same change. The hidden `?debug=1` aside was the only previous runtime dev surface; with it gone, `?debug=1` is unused and the URL-param read at `src/spa/routes/game.ts:1137` is removed. If any external tooling (e2e tests, playtest scripts, scripts in `scripts/`) sets `?debug=1`, those callers update or drop the param.
- The in-flight pip requires request-start / first-token / completion / error edges from the LLM provider — surface that does not exist today (`browser-llm-provider.ts` exposes `onUsage` callbacks only, no lifecycle edges). Because the consumer is `__DEV__`-gated, the new observability surface can also be `__DEV__`-gated: a lifecycle event emitter that is a no-op shim in prod and a real EventTarget in dev. The provider doesn't pay a prod-runtime cost for the new edges.
- New contributors will not discover the inspector by typing a flag in a live URL; they discover it by running a dev build, which is the population that needs it. The README and any onboarding docs should mention `pnpm run dev` opens the inspector by default, rather than documenting a flag that doesn't exist.
