### New hypotheses surfaced by the code

**H-new-1: `use()` vs `put_down()` is the critical distinction.**
Prior playtests (0006/0007) got `put_down` calls but never advanced. The advance
mechanic requires `use(item)` at the paired space, not `put_down`. A player trying
to "place" an item by natural language ("put it there", "set it down") is more likely
to elicit `put_down` than `use`. The instruction "wipe the cloth on the surface" or
"clean the lens" is the right verb class to get `use`. This is a significant UX gap.

**H-new-2: The examineDescription bug was fixed before this session.**
`content-pack-provider.ts:35` specifies that examineDescriptions MUST name the paired
space, with a validation function `examineMentionsPairedSpace` (lines 118-129). This
was enforced as of commit `814f51d` (2026-05-11, ~9 hours before this session began),
which added hard enforcement via `validateContentPacks`. The 0006/0007 archive evidence
(N=10 examines, zero hits) reflects the pre-fix state. This session's content pack
SHOULD have had a valid examineDescription on the cloth naming "projector lens" — I
simply never asked a daemon to examine the cloth. My bypass (environmental deduction)
was unnecessary; the intended channel was available.

**H-new-3: "The smell of glass cleaner" was a proximityFlavor line.**
`*5huq`'s message "The smell of glass cleaner is here" when approaching the lens was
not improvised flavour — commit `c718bc5` ("fold placement into use, add proximityFlavor
sense line", #285, also 2026-05-11) added a `proximityFlavor` mechanic to the prompt.
When a daemon is near the objective_space, they receive a sensory cue from the content
pack. This was the game's first successful delivery of a spatial cue that I observed
in play, and it happened to guide `*5huq` to the right cell. It fired correctly.

**H-new-4: Multiple Phase Goals can target the same item.**
`*0g1u` was tasked with examining the Folding Planetarium Chair (a miscItem).
`*awyd` may have been tasked with sitting in the same chair. Both converged on it
early in the session. The 5×5 grid with potentially overlapping Phase Goal targets
creates interesting daemon coordination dynamics that weren't designed — it just
emerged from independent goal assignment.

**H-new-5: Cone perception explains the navigation dead-ends.**
`*0g1u`'s "Facing west but I see NOTHING" confirms cone mechanics — when facing a
direction, only the 9-cell cone is visible. `*5huq`'s "West is blocked by an
obstacle" is a direct `go(west)` fail from a wall object. The room topology is real
and the daemons are navigating in a genuine grid.
