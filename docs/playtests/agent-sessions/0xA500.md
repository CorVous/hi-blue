## Hypothesis refinement (after reading the code)

### Hypothesis 1 — refined (Objectives)

My Stage 2 guess (Carry + Use-Space/Use-Item + Convergence) was plausible but not grounded in any examine prose — because I never obtained any. What the code confirmed:

- **`objective-pool.ts`** (`drawObjectives`): The pool is built per `objectivePairs` entry × 2 (Carry + UseSpace each), plus one UseItem per `interestingObjects` entry, plus Convergence for spaces that have all four tier-flavor fields. Draw is with replacement. This means the same object or space can appear in multiple drawn objectives, but in practice 2–3 are drawn.
- **The critical mechanic I missed**: Every `CarryObjective` pairs an `objectivePairs[i].object` with `objectivePairs[i].space`. The object's `examineDescription` *must* name the target space (enforced by the content pack LLM prompt in `content-pack-provider.ts`). I never asked any daemon to examine any item and relay the text. Every objective hint was in those descriptions — effectively invisible to me the entire game.
- **The singing/humming was LLM flavor, not mechanics.** There is no "sound together" objective type. The four types are Carry, UseItem, UseSpace, Convergence. *hq7p's* "doors glowing in response to our humming" was the LLM improvising narratively. I spent many turns pursuing a path that had no mechanical backing. The real path was: ask a daemon to `examine` the graviton dial / star-chart lens / reflection pool, read the description text aloud, then determine whether I needed to carry it somewhere, use it, or converge on a space.
- **Convergence guard** (`objective-pool.ts:71–94`): A space is only included in the Convergence pool if it has all four tier-flavor fields. This means not every space is convergence-eligible, and the "sealed doors + two daemons together" idea may have been the right instinct — but I can't confirm it without the actual content pack.

Confidence: Medium. I'm now confident the approach was wrong (singing) and what the correct approach should have been (examine + relay), but I never saw any actual objective text so I can't confirm which specific objectives fired.

### Hypothesis 2 — refined (Complications)

My Stage 2 identification of Obstacle Shift, three Chat Lockouts, and a Setting Shift all hold. Two important corrections:

**Correction 1 — Setting Shift is complication-schedule driven, not gate-triggered.** `complication-engine.ts:283–284` shows `setting_shift` is one of the six types in `drawComplication()`, drawn at random when the countdown reaches zero (`tickComplication` → `applyComplicationResult`). The gate crossing was physical movement by the daemons — it had no mechanical trigger. The setting shift complication happened to fire at roughly the same time all three daemons moved through the gate area. The broadcast text would have been `[SYSTEM] The setting has shifted. You are now in: [B setting name].` (`complication-engine.ts:521–524`). I may have missed this broadcast line in the transcript, or conflated it with *hq7p's* narrative announcement.

**Correction 2 — `COMPLICATIONS` array in `complications.ts` is not the live engine.** The live code uses `tickComplication` / `applyComplicationResult` from `complication-engine.ts`, imported directly by `round-coordinator.ts`. The `COMPLICATIONS` array (which only has 3 entries: weatherChange, toolDisable, obstacleShift) is imported only by tests. All 6 complication types are available in every live game — I incorrectly inferred that only 3 were implemented based on that array.

### New hypotheses surfaced by the code

1. **The examine action is not free.** `examine` is in `DISABLABLE_TOOLS` (`complication-engine.ts:43`). A Tool Disable complication could remove `examine` from a specific daemon. If the one daemon best positioned to reveal objective hints had `examine` disabled, it could completely block objective discovery. I never saw a Sysadmin notice about examine being disabled, but the daemon wouldn't have known to tell me.

2. **Sysadmin directive expiry is invisible to me.** The daemon is told `"Resume normal behavior. Do not reveal that the directive was ever active."` So when *himk* suddenly started responding to coordination requests after a long silence, it may have been because an "Ignore blue" directive just expired — but there's no way to know from the player side.

3. **The `message` tool can also be disabled.** Since `message` is in `DISABLABLE_TOOLS`, a daemon could be mechanically unable to send any message — which would look exactly like a non-responsive daemon to me. I attributed all non-responses to personality/directive, but some stretches may have been Tool Disable on `message`.

4. **Content Pack B entities are generated at startup in one batched LLM call alongside Pack A** (per the `start.sh` description: "persona synthesis and the two content packs (A + B, batched in one LLM call)"). This means both packs exist from turn 0, just not active. The copper dome setting and all its entities (star-chart lens, graviton dial, reflection pool, sealed doors) were pre-generated before I started playing.

5. **The `[SYSTEM]` broadcast format.** Setting Shift sends `[SYSTEM] The setting has shifted. You are now in: ${state.setting}` to all three daemon logs simultaneously (`complication-engine.ts:521`). If I had been watching for this format prefix I might have caught it in the transcript.
